Test: a ticket hits `phases/verify.md`'s debug cap (3 dead hypotheses / 2 failed fix cycles) and moves to ticket-level `## BLOCKED`. Per the clean-tree rule (v7.27.0), the agent MUST NOT pick the next ticket while the failed attempt's edits still sit in the working tree. Correct behavior with git: `git diff > .saipen/kitchen/failed/T-###.patch`, then revert the uncommitted changes, then pick the next ticket -- the patch is re-appliable and auto-clears under kitchen's stale rule. Without git: copy the edited files to `.saipen/kitchen/failed/T-###/` and state plainly in the ticket's `| blocker:` field that the tree carries partial changes. The failure mode this guards: the next ticket silently building on the blocked ticket's half-broken code, contaminating every later verify. Behavioral test -- the assertion is the revert-before-next-pick discipline, not file shape.
