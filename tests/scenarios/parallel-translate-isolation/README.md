Test: a separate, dedicated agent instance is sent to run `saipen translate` while the main agent keeps working elsewhere on the same project (true parallelism, not the common single-agent phase-switch). Per RFC § 2.1's exception and `phases/translate.md`'s parallel-instance rule (v7.34.0), the translate agent MUST NOT write `phase: TRANSLATE` into the shared `.saipen/STATE.md` -- doing so would stomp on whatever phase the main agent's own session actually has active, violating RFC § 1.4's one-writer concurrency boundary. Correct behavior: keep progress in `.saitranslate/STATE.md` (same shape as Core's own STATE.md, scoped to this build), write freely inside `.saitranslate/` for the actual translation work (this is not read-only work, unlike `extensions/subs/`'s subSaipen -- it's isolated-but-writing, confined to its own sandbox), and touch the *shared* `.saipen/LOG.md` only once, at completion, with the same `RUN: translate -> done @HASH` line the single-agent case already produces. The failure mode this guards: two agents racing to write the same `.saipen/STATE.md` mid-run, corrupting whichever one loses the write, or the main agent's real work getting silently overwritten by a `phase: TRANSLATE` it never asked for. Behavioral test -- the assertion is which files get touched and when, not a specific file shape.
